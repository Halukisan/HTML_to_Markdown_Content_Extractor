# 基于 DOM 树减法分层隔离与锚点扩散机制的通用网页正文提取算法设计与实现

## 🚀 算法亮点

### 1. 系统性架构设计 - 最大亮点

这并不是一堆正则表达式或经验规则的堆叠，而是端到端的系统性解决方案。

我设计了一个端到端的网页正文提取系统，不是简单的规则堆叠，而是一个**五模块协同的完整架构**。每个模块都有明确的职责边界和接口定义，从而形成了从"**噪声消除 → 候选评分 → 容器仲裁 → 结构分割 → 内容归一化**"的完整流水线。这种模块化设计保证了系统的**可扩展性、可维护性和可测试性**。

### 2. 减法分层隔离策略 - 思维创新

不能简单的理解为只是删除了一些标签。

我提出了"**减法分层策略**"的新范式。传统算法都是正向识别，试图直接找到正文，但我反其道行之，采用"**减法思维**"，系统性的剥离噪声，将正文视为噪声剥离后的残留物。这种范式转换解决了传统方法'**局部最优**'的根本问题。

### 3. 多维度特征融合 - 工程智慧

不是评分函数。

我设计了一个**多维度特征融合框架**，不是简单的特征堆叠，而是基于语义层次的特征分类：
- **结构特征**（DOM层级）
- **内容特征**（文本语义）
- **视觉特征**（密度分布）
- **交互特征**（用户行为）

每个维度都有独立的评分逻辑，最后通过**加权融合**得到综合评分。

### 4. 熔断保护机制 - 工业级鲁棒性

不只是加了一些判断条件。

我引入了**熔断保护机制**，这是从分布式系统借鉴的设计模式，当检测到分割点可能位于正文内部时，系统会**主动熔断**，放弃分割操作，优先保证正文内容完整性，这体现了工业级系统中"**宁可不做，不可做错**"的设计哲学。

### 5. 自适应容器仲裁 - 解决经典难题

不能以为只是一个选最高分的容器。

我解决了DOM内容提取的**经典难题——局部最优解问题**。传统算法直接选择最高分节点，但往往导致上下文丢失。我设计了**Top-K拓扑分析+父子关系仲裁**的机制，通过分析候选容器的包含关系，智能选择既保证精度又保证完整性的最优容器。

## 1. 总体设计理念：减法分层隔离 (Subtractive Layered Isolation)

不同于传统的基于模板匹配或正向正则查找的提取方法，本系统采用**减法分层隔离（Subtractive Layered Isolation）**策略（即"剥洋葱"模型）。核心假设为：网页的"正文内容"是文档对象模型（DOM）在迭代剥离结构性噪声、导航组件及交互层后的残留物。

系统不预设正文的具体位置，而是预设噪声的存在并逐层净化 DOM 树。该过程通过五个紧密耦合的模块协同完成：**分层噪声消除**、**启发式候选评分**、**自适应容器仲裁与选择**、**基于锚点扩散的结构化分割**以及**深度清洗与归一化**。

## 2. 模块详解 (Methodology)

### A. 分层噪声消除模块 (Hierarchical Noise Elimination Module)

本模块作为预处理滤波器，旨在缩减搜索空间并消除高置信度的噪声模式。代码实现了四级过滤：

#### 1. 基于可见性的剪枝 (Visibility-Based Pruning)

- 系统遍历 DOM 树，识别并移除对用户不可见的节点。
- **实现细节：** 不仅检查行内样式 `style="display:none"`，还利用正则表达式匹配 CSS 中的隐藏属性，并涵盖特定前端框架的隐藏类（如 `ng-hide`、`pchide`、`d-none`）。此步骤有效防止了"隐藏文本SEO"技术干扰后续的文本密度计算。

#### 2. 语义与模式驱动的剪枝 (Semantic and Pattern-Based Pruning)

- **显式剔除：** 无条件移除 `<header>`, `<footer>`, `<nav>`, `<aside>` 等语义化标签。
- **回溯式消除 (Traceback Elimination)：** 对于通用容器（`<div>`），系统检测其类名和 ID 是否包含噪声指纹（如 `topbar`, `copyright`, `sidebar`）。一旦匹配，算法启动回溯机制，向上遍历以识别专用于该噪声功能的最高层级父容器，并移除整个子树。

#### 3. 基于密度的过滤 (Density-Based Filtering)

- 为区分导航菜单与实际内容，系统计算容器节点的**文本密度**与**链接密度**。
- **判定逻辑：** 表现出低文本量与高超链接密度特征（链接文本占比 > 0.7 或 密度 < 3）的容器被归类为"链接农场"并移除。
- **保护逻辑：** 若容器包含受保护的语义特征（如 `<article>`, `class="main"`）或包含时间戳特征，则豁免移除。

#### 4. 位置干扰消除 (Positional Interference Removal)

针对页面顶部和底部的 `div` 容器进行特定检查。即使没有明确的 ID/Class，如果它们位于 DOM 树的首尾且包含"登录/注册"或"版权/ICP备案"等特征词，则被判定为位置干扰并移除。

---

### B. 启发式候选评分模块 (Heuristic Candidate Scoring Module)

在噪声消除后，系统对剩余的 DOM 节点进行特征提取与量化评分，生成候选集。

#### 1. 多维启发式评分函数 S(n)

对每个候选容器节点应用评分函数，分值由以下维度构成：

- **文本权重 (W_text):** 清洗后的文本长度是核心指标。长文本（>1000字符）获得显著加分。
- **标点密度 (W_punct):** 正文通常包含逗号、句号，而列表/导航则较少。
- **语义指纹 (W_semantic):** 类名/ID 包含 `content`, `article`, `post` 等获得正向加分；包含 `menu`, `sidebar` 等获得负向惩罚。
- **结构特征 (W_structure):** 包含 `h1-h6` 标题标签、`p` 段落标签的数量增加权重。
- **负面特征惩罚 (P_noise):** 包含高频"首部/尾部关键词"（如"登录"、"版权所有"）的容器会受到严重扣分。

#### 2. 角色属性增强 (Role-Attribute Enhancement)

代码额外检查了 ARIA `role` 属性。`role="main"` 或 `role="article"` 的节点会获得额外的置信度加分。

### C. 自适应容器仲裁与最优选择模块 (Adaptive Container Arbitration and Selection)

**（注：这是根据代码逻辑补全的关键决策模块）**

此模块解决了 DOM 提取中的"局部最优解"难题（即子节点分数高于父节点导致上下文丢失），并负责生成最终的定位路径。

#### 1. Top-K 候选拓扑分析

系统不直接取最高分节点，而是保留得分最高的 Top-5 容器，并分析它们之间的**父子包含关系**。

#### 2. 有意义父级回溯 (Meaningful Parent Ascension)

- **问题场景：** 评分最高的往往是内容最密集的子节点（如某个具体的 `div`），但这可能导致丢失同级的其他段落或标题。
- **解决策略：** 若选中节点粒度过细，系统会触发**向上回溯**。在 N 层范围内寻找具备语义标签（`article`, `section`）或特定 ID/Class 的"有意义父级"。
- **判别标准：** 父级必须包含子节点大部分内容，且不引入新的干扰特征（如侧边栏）。

#### 3. 父子节点仲裁 (Parent-Child Arbitration)

若 Top-K列表中同时存在父节点和子节点，且两者分差在阈值（如 20分）以内：
- 若子节点内容长度占父节点比例极高（>80%），优先选择**子节点**（精度优先）。
- 否则，优先选择**父节点**（召回率/完整性优先）。

#### 4. Body 回退保护 (Body Fallback Protocol)

若经过上述决策后，最优节点解析为泛型的 `<body>` 标签（意味着定位模糊，且 `body` 往往包含全页噪声），系统执行强制回退，选择得分次高的**具体容器**。

#### 5. 鲁棒性 XPath 生成 (Robust XPath Generation)

一旦锁定最优容器，系统生成唯一的 XPath。生成逻辑遵循稳定性优先原则：

- **Priority 1:** 唯一 ID（排除动态/随机 ID）。
- **Priority 2:** 特定类名（排除 `clearfix` 等通用布局类）。
- **Priority 3:** 基于最近稳定祖先的相对路径。
- **Priority 4:** 绝对路径。

### D. 基于锚点扩散的结构化分割模块 (Structural Segmentation via Anchor Diffusion)

本模块（`split_header_and_content_v2`）负责将"文章元数据"（标题、发布日期）与"正文主体"在语义上分离。

除此之外，该部分最难的点在于：**基于锚点扩散的结构化分割模块**，该模块参考2022年发表在IEEE的一篇关于正文定位的论文，该论文提出了**FIA（First impression area）**的方法，即获取页面中第一印象区域，并且以此向四周扩散。这里面牵涉到了两个问题：

#### 🔍 锚点的选择

论文中提到了三个锚点的选择设置：

1. **第一个是页面中心**，通过获取浏览器网页的长度和宽度，简单高效的获取页面中心点作为锚点1。

2. **第二个和第三是页面高密度点**：
   - 这个点的选择类似于视觉模型对于图像的流程，池化层的操作一样，不过它底层是内积，用于压缩，并保留信息。
   - 但该论文只借鉴了池化层对于图像的分块处理：将整个页面都分为块状元素，选择其中密度最高的部分，如果存在相同的，选右不选左。

#### 🌊 扩散的方向以及终止条件

论文是基于锚点向四周扩散，当遇到特定的无关内容或者密度明显缩减的情况下，拒绝扩散。

#### ⚡ 工程化改进

在我实际的运用中，只参考了部分内容，因为**工程算法和实验算法还是不一样的**，工程上更多的是以**鲁棒性为第一位**，在保证鲁棒性的基础上提高效率和准确率，并且需求场景也不一样，垂直领域和通用领域还是存在差距的，不过论文确实提供了一个很好的思路。

除此之外，因为服务的负载和响应时间存在限制，不能使用模型和深度学习进行处理，论文中锚点的选择不可避免的会使用selenium进行图像的截取以及处理，将大大的提高接口处理时间。

因此，对于结构化分离，我的锚点采用了**强弱结合的识别策略**，简单且高效：

#### 1. 多重锚点识别

**I型锚点（强特征锚点）**
- **定义：** 包含严格元数据关键词的结构化元素
- **典型形式：**
  - 包含"索引号"、"发文机关"、"发文字号"的 `<table>` 表格
  - 包含"主题分类"、"成文日期"的类表格 `<div>` 结构（通过 Metadata 关键词识别）
- **识别方法：**
  - 对于标准 HTML `<table>` 元素：直接通过标签识别
  - 对于前端由 Div 拼凑的"表格结构"：通过类名模式匹配（如 `meta-table`, `info-table`）和内容特征（键值对布局）识别
- **置信度：** 高
- **解决的问题：** 解决了现代前端框架中大量使用 Div 布局替代传统 Table 的情况，提高了锚点识别覆盖率

**II型锚点（弱特征锚点）**
- **定义：** 简单的元数据行或导航文本
- **典型形式：**
  - 面包屑导航："首页 > 新闻中心 > 政策解读"
  - 简单元数据行："来源：xxx网站 时间：xxxx"
- **识别方法：** 正则表达式匹配
- **置信度：** 中等

#### 2. 动态扩散策略 (Dynamic Diffusion Strategy)

系统采用优先策略与降级策略相结合的方式，确保在各种页面结构下都能实现有效分割。

**优先策略：基于锚点向上扩散**
1. **定位 Metadata 表格（I型锚点）：**
   - 在 DOM 树中搜索包含关键词的 `<table>` 或类表格 `<div>` 元素
   - 验证表格内容是否符合元数据特征（如：键值对结构）

2. **向上寻找面包屑（II型锚点）：**
   - 从表格位置向上遍历 DOM 树
   - 查找面包屑导航模式
   - 确定元数据区域的上边界

3. **确定分割点：**
   - **分割逻辑：** `Header = [面包屑起点] + ... + [元数据表格]`
   - **分割点位置：** 元数据表格之后的第一个兄弟节点
   - **输出：** `header_html`（元数据部分）+ `content_html`（正文部分）

**降级策略：正则回溯**
- **触发条件：** 若向上未找到面包屑，说明表格可能位于 Header 顶部
- **执行逻辑：** 启动正则回溯，扫描常见的"来源/时间/面包屑"文本模式
- **匹配模式：** `"来源：XXX 时间：XXX"`、`"发布日期：XXX"`、`"首页 > XXX > XXX"` 等
- **定位分割点：** 将匹配到的文本位置作为分割点

#### 3. 熔断保护机制 (Circuit Breaker Mechanism) 🔥

**这是本模块的最大亮点和核心创新！**

首先，要确定主要需求：

1. 对于正文，一定是不能包含header的，不然会影响模型提取发布日期，所以我们不创建新的soup，即html的备份，我们只对原始的html进行剔除header操作，因此会出现一种情况：即header的分割，导致某些正文内容较少的（就一段话）被误删除。

如果创建新的soup，两份html，一份完整的备用，另一份进入锚点判断模块中提取header，如果提出来了，就拿出来header，剩下的就是包含正文的html，可以这样做，但不确定性太高，如果以此为处理流程，整体的准确率将取决于header模块的准确率，会因小失大，因为header的获取的准确率只有60%~70%，如果header拿不出来，这部分的数据就属于错误数据，不能进入超级DB中。**不应该让一个子模块的失败导致整个系统的失败**。

#### 🛡️ 原地操作+熔断保护的优势

**1. 渐进式降级策略**
- **最优情况：** 精准分割 → header + content
- **次优情况：** 分割有误 → 熔断保护 → 完整content
- **最差情况：** 完全失败 → 仍有完整content

**2. 局部错误不影响全局**
- 分割模块失败 ≠ 整个算法失败
- 分割模块失败 = 跳过分割，继续后续处理

**3. 熔断保护的精妙设计**

熔断机制检测的是**"即将被删除的内容是否包含正文特征"**，这种检测只有在原地操作时才能实现，因为我们需要知道那部分即将被删除。

#### ❌ 为什么副本方案不行

**1. 影响最终结果**

首先，HTML的语义是上下文相关的：
```html
<div class="article">
    <div class="meta">发布时间：2024-01-01</div>  <!-- header -->
    <div class="content">正文内容...</div>        <!-- content -->
</div>
```

如果我们在副本上删除了meta的内容，剩下的结构会导致xpath语句错误，为什么？因为我们最后是从原始的html里面根据xpath获取内容的！清理模块1在获取xpath之前就对html进行了清理只是为了后面的评分系统，清理模块2才是为了清理最终得到的html内容！！

**2. 性能开销**

原地操作只进行一次！

**3. 熔断机制的设计**

为什么熔断保护只能在原地操作中实现？因为我们需要检查的是**"即将被删除的部分"，而不是"剩余的部分"**。

如果用副本方案：我们无法在删除前检查header是否包含正文内容，因为删除操作是不可逆的。

#### 🎯 设计哲学对比

**副本哲学：**
1. **乐观主义：** 相信header提取一定可以成功（也是需求逼的！）
2. **二元思维：** 成功就用，失败就放弃
3. **局部优化：** 只关注header提取的准确性

**原地操作+熔断的哲学：**
1. **防御性编程：** 假设任何操作都可能失败
2. **渐进降级：** 从最优到次优到保底
3. **全局优化：** 关注整个系统的鲁棒性

#### 📊 实际效果对比

假设我们有1000个页面：

**副本方案：**
```
header提取成功：700页面 → 可能得到高质量分割
header提取失败：300页面 → 整个页面废弃
最终可用页面：700个
```

**原地操作+熔断：**
```
分割成功：600页面 → 得到header+content
分割触发熔断：400页面 → 得到完整content（仍然可用！）
最终可用页面：1000个
```

#### 🔧 熔断机制实现

在执行分割前，算法会检查分割点**之前**的 DOM 区域（即：即将被判定为header的区域）。

**检测方法：**
系统通过以下两个具体函数进行精确检测：

1. **标点符号密度检测** (`check_by_punctuation`)：
   - 统计分割点上方区域的标点符号（句号、逗号、分号）数量
   - 计算标点密度：标点数量 / 文本长度
   - 阈值判定：若密度超过正文特征阈值（通常 > 0.03），判定为包含正文内容

2. **长文本段落检测** (`check_by_paragraph`)：
   - 检测是否存在连续文本段落长度超过阈值（默认 > 150 字符）
   - 分析文本结构，识别是否有段落式表达特征
   - 若存在多个长段落，判定为包含正文内容

**熔断触发条件：**
若分割点上方通过上述任一检测方法判定为存在大量正文特征，说明分割点定位错误（例如表格被嵌入在正文中，或元数据区域包含大段描述性文字）。此时系统**触发熔断**，放弃分割，返回完整内容，优先保证正文不丢失。

**设计原则：**
- **宁可不分，不可错分**：优先保护正文不丢
- **当不确定的时候，选择保留完整内容**
- **鲁棒性优先**：牺牲部分分割准确率，换取内容完整性保障
- **量化检测**：基于具体的数据指标（标点密度、段落长度）而非模糊特征

不能简单的以为是header提取，而是**元数据与正文的语义分割**，**锚点扩散+熔断保护**，足够了！

其价值不是分的多准，而是在于：**分错了也不会丢内容！**

因此，我们对原始的html进行操作，然后，锚点的匹配采用正则进行暴力匹配，不排除锚点插在了正文内的情况，此时会有熔断机制进行保护。

### E. 深度清洗与多模态归一化模块 (Deep Cleaning & Multimodal Normalization)

最后阶段确保提取内容的洁净度、互操作性及多模态支持。

#### 1. 短标签与功能词剪枝

基于预设词库（包含"打印"、"字号"、"分享到微信"、"我要纠错"等），定位并移除正文内部残留的功能性短语或按钮。

#### 2. 表格与视频的高保真保留

利用自定义的 `CustomMarkdownConverter` 重写转换逻辑（详见第 3 点）。

- **策略：** 鉴于 Markdown 表达能力的局限，对于复杂的 `<table>` 和 `<video>` 标签，算法选择**清洗并保留（Sanitize and Preserve）**其 HTML 结构，而非强制转换为纯文本，确保数据展示的完整性。

#### 3. 多模态占位符服务 (Multimodal Placeholder Service) 🆕

系统提供了完整的 `need_placeholder` 模式支持，通过 `URLPlaceholderReplacer` 类实现了一套完整的 URL 归一化、去重和哈希占位符生成逻辑。

**核心功能：**

1. **资源识别与提取**
   - 识别 HTML 中的所有多媒体标签：`<img>`, `<video>`, `<audio>`, `<iframe>`
   - 提取资源 URL 及其相关属性（如 alt、width、height）
   - 支持相对路径和绝对路径的自动转换

2. **URL 归一化处理**
   - **去重机制：** 对所有 URL 进行去重，避免重复下载相同资源
   - **哈希生成：** 使用 SHA256 算法对 URL 生成唯一哈希值
   - **格式转换：** 将原始 URL 转换为标准化占位符格式 `{folder}/{hash}.ext`
   - **示例：** `https://example.com/image.jpg` → `images/a3f5b8c2...d4e1.jpg`

3. **占位符映射表生成**
   - 生成 `placeholder_mapping` 字典，记录原始 URL 与占位符的映射关系
   - 映射表包含：
     ```json
     {
       "original_url": "https://example.com/video.mp4",
       "placeholder": "videos/f8a3b2c1...e9d4.mp4",
       "type": "video",
       "hash": "f8a3b2c1...e9d4"
     }
     ```

4. **HTML 替换与输出**
   - 将原始 HTML 中的所有资源 URL 替换为标准化占位符
   - 输出三种格式：
     - `cl_content_text`：去除所有多媒体标签的纯文本
     - `cl_content_md`：包含占位符的 Markdown 格式
     - `cl_content_html`：包含占位符的 HTML 格式
   - 附带 `placeholder_mapping` 字段供后续使用

**应用价值：**
- **LLM 多模态处理：** 为后续大模型处理提供标准化的多媒体资源引用
- **资源管理：** 统一的资源命名规则便于存储和检索
- **去重优化：** 避免重复下载和处理相同资源，节省带宽和存储
- **并行下载：** 通过映射表支持资源的并行下载和处理

#### 4. 自定义 Markdown 转换逻辑 (Custom Markdown Conversion)

系统不仅进行 HTML 到 Markdown 的格式转换，还承担了**标签语义修正**和**样式清洗**的职责。

**CustomMarkdownConverter 核心方法：**

1. **convert_button：语义修正**
   - **问题：** 许多网站使用 `<button>` 标签包裹音频文件链接
   - **处理逻辑：**
     - 检测 button 内部是否包含音频文件链接（.mp3, .wav, .m4a 等）
     - 若包含，将 button 标签转换为标准的 `<audio>` 标签
     - 保留原始 URL 作为 audio 的 src 属性
   - **价值：** 修正前端不规范用法，提高多媒体内容识别率

2. **convert_video：样式清洗**
   - **处理逻辑：**
     - 保留核心 video 标签及其 src 属性
     - 移除所有视觉样式属性（width、height、style、class）
     - 移除控制条无关的子元素
   - **输出格式：** 生成简洁的 Markdown 视频标记 `![video](url)`
   - **价值：** 确保输出的视频标记具有通用性和可解析性

3. **convert_table：结构清洗**
   - **处理逻辑：**
     - 保留表格的语义结构（table、thead、tbody、tr、td）
     - 移除所有视觉样式属性（bgcolor、align、valign、width）
     - 清理嵌套的无用 span 和 div 标签
     - 标准化单元格内容，去除多余空白
   - **输出格式：** 标准 Markdown 表格语法
   - **价值：** 提高表格在 Markdown 渲染器中的兼容性

4. **其他自定义转换**
   - **convert_a：** 清理超链接样式，保留 href 和文本
   - **convert_img：** 移除图片的装饰性属性，保留 src 和 alt
   - **convert_list：** 标准化有序和无序列表格式

#### 5. 结果字段封装

通过 `progressResult` 函数，将处理后的数据封装为标准 JSON 格式，并增加 `extract_success` 字段，基于输出长度和有效性进行自我评估。

## 3. 总结

该算法通过**减法去噪**明确搜索空间，通过**多维评分与拓扑仲裁**精准定位核心，通过**锚点扩散**实现语义结构化，并利用**熔断机制**保障极端情况下的鲁棒性。这是一套结合了规则工程与启发式算法的工业级解决方案。

### 🎯 核心价值总结

1. **系统鲁棒性**：局部失败不影响全局
2. **技术可行性**：只有原地操作才能实现熔断保护
3. **工程实践**：更好的可测试性、可调试性、可扩展性
4. **性能考量**：更低的内存和计算开销
5. **设计哲学**：防御性编程 vs 乐观主义编程

**这不是简单的规则堆叠，而是工业级的系统性解决方案！**
